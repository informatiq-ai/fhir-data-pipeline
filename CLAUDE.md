## Project Overview

`fhir-data-pipeline` is a vendor-agnostic reference architecture for multi-tenant
clinical data normalization at state HIE / Health Data Utility scale. It demonstrates
a medallion-style Bronze → Silver → Gold pipeline with synthetic data as proof of concept.

**Purpose:** Portfolio artifact demonstrating healthcare data architecture competency.
**Audience:** Healthcare data and IT professionals, potential collaborators, GitHub community.
**Author:** Phil Babbel · informatiq.ai

## CI/CD Architecture

This repository uses GitHub Actions for continuous integration and automated results
regeneration. There are two workflows. Both run from the repo root with no external
dependencies beyond the packages in the install step.

### Workflow 1: CI (`ci.yml`)

**Trigger:** Every push to `main` or `develop`, every pull request to `main`.

**What it does:**
1. Runs `python run_pipeline.py --tests-only` — full pytest suite (41 tests)
2. Runs `python run_pipeline.py --no-write` — all 5 pipeline stages against synthetic data
3. Uploads `PIPELINE_RESULTS.md` as a build artifact

**Matrix:** Python 3.11 and 3.12. Both must pass.

**Purpose:** Gate merges on test passage. Catch regressions in ingestion, normalization,
identity resolution, and analytics logic before they reach main.

### Workflow 2: Regenerate Results (`regenerate-results.yml`)

**Trigger:** Push to `main` when pipeline-relevant files change (ingestion/, transforms/,
schema/, data/synthetic/, tests/, run_pipeline.py). Also triggerable manually via
`workflow_dispatch` from the GitHub Actions UI.

**What it does:**
1. Runs `python run_pipeline.py` — full pipeline + tests, writes `PIPELINE_RESULTS.md`
2. Commits the updated `PIPELINE_RESULTS.md` back to main with `[skip ci]` to prevent
   an infinite trigger loop

**Purpose:** Keep `PIPELINE_RESULTS.md` current in the repo so GitHub visitors see
live output without having to run the pipeline themselves.

**Requirement:** Repo must have Actions write permissions enabled.
Settings → Actions → General → Workflow permissions → "Read and write permissions."

## Testing Strategy

Tests live in `tests/test_transforms.py` and are organized into six classes.
All 41 tests run against synthetic data in `data/synthetic/` — no database,
no cloud credentials, no environment variables required.

### Test Classes

| Class | Tests | What It Covers |
|---|---|---|
| `TestHL7Timestamp` | 5 | HL7 timestamp parsing, timezone stripping, edge cases |
| `TestHL7Parser` | 9 | ADT/ORU parsing, malformed messages, batch processing, tenant fallback |
| `TestFHIRIngester` | 8 | Bundle splitting, tenant extraction, raw payload integrity, error records |
| `TestMPIIndex` | 6 | UMPI minting, deterministic matching hierarchy, cross-facility linkage |
| `TestTerminologyService` | 8 | LOINC/RxNorm/SNOMED lookups, case insensitivity, unmapped handling |
| `TestNormalizeFHIRObservation` | 5 | SOURCE_LOINC path, terminology fallback, unmapped audit log |

### Test Types

**Unit tests:** All 41 current tests are unit tests. Each test exercises a single
function or class method in isolation using synthetic data fixtures.

**Integration tests:** Not yet implemented. The intended integration test layer would
wire ingestion → identity resolution → normalization → Gold construction as a single
end-to-end pass using the synthetic data files, asserting on the final Gold record
outputs rather than intermediate structures. Add these to `tests/test_integration.py`.

**Deployment:** Not yet implemented. The current stack is local/reference only.
A deployment pipeline would target a Databricks or Snowflake environment and would
require secrets management (stored in GitHub Secrets, never in code) and environment-
specific configuration. Add deployment workflows to `.github/workflows/deploy.yml`
when a target environment is defined.

### Running Tests

```bash
# Full test suite
python -m pytest tests/ -v

# Via runner (same result, explicit)
python run_pipeline.py --tests-only

# Via Makefile
make test
```

## Pipeline Runner

`run_pipeline.py` is the single entry point for running the full pipeline
and regenerating `PIPELINE_RESULTS.md`.

```bash
python run_pipeline.py              # Run all stages + tests, write PIPELINE_RESULTS.md
python run_pipeline.py --no-write   # Run all stages + tests, print only
python run_pipeline.py --tests-only # Run pytest suite only
make run                            # Alias for python run_pipeline.py
make test                           # Alias for --tests-only
```

**Design:** Each pipeline stage script (`hl7_parser.py`, `fhir_ingester.py`, etc.)
has a `if __name__ == "__main__":` demo block that `run_pipeline.py` invokes as
a subprocess. Output is captured, checked for non-zero return codes, and written
to `PIPELINE_RESULTS.md` with contextual annotations.

## Development Rules for Claude Code Sessions

### Before writing any code
1. Read this file in full.
2. Read `PIPELINE_RESULTS.md` to understand the current passing state.
3. Run `python run_pipeline.py --tests-only` to confirm baseline.

### One file per response
Do not modify more than one file per Claude Code response unless explicitly
instructed. Commit between changes.

### Commit discipline
Commit after every working change. Never leave the repo in a broken state between
commits. Commit messages follow: `type: description` (e.g., `fix:`, `feat:`, `docs:`, `chore:`).

### After any code change
1. Run `python -m pytest tests/ -v` — all 41 tests must pass before committing.
2. Run `python run_pipeline.py` to regenerate `PIPELINE_RESULTS.md`.
3. Commit both the code change and the updated `PIPELINE_RESULTS.md` together.

### No silent failures
Every ingestion error lands in Bronze with `processing_status=ERROR`.
Every unmapped terminology code produces an UNMAPPED entry in `silver.normalization_log`.
Nothing is dropped silently at any layer.

### Terminology mapping is deterministic
LOINC, RxNorm, and SNOMED mappings are table lookups from authoritative sources.
Do not use model inference for structured code mapping. LLMs are appropriate only
for unstructured text extraction.

### No real data — ever
`data/synthetic/` contains only fictional patients, MRNs, and clinical values.
Do not introduce real patient data, real MRNs, real provider NPIs, or any PHI
into this repository under any circumstances.

## Known Gaps (Intentional Scope Limits)

These are documented omissions, not bugs. Each is an extension point for future sessions.

- **Probabilistic MPI:** Deterministic matching only. A production HIE needs a
  referential matching layer (Verato or Fellegi-Sunter) on top of the current hierarchy.
- **Integration tests:** Unit tests only. `tests/test_integration.py` does not yet exist.
- **Deployment pipeline:** No `.github/workflows/deploy.yml`. No target environment defined.
- **Elixhauser Index:** `silver_to_gold.py` calculates Charlson only. Elixhauser is
  scaffolded (`elixhauser_index=0`) but not implemented.
- **Procedures and Immunizations:** Schema extension points documented in
  `docs/canonical-data-model.md`. Tables not yet created.
- **Terminology server integration:** Static lookup tables in `TerminologyService`.
  Production replacement is a call to NLM VSAC API or a hosted FHIR TerminologyServer.
- **PHI encryption at rest:** Assumed platform-level. Not implemented in this reference.
